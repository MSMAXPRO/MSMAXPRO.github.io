Chapter 2: Intro to Node.js & Express

Backend Developer Roadmap

Alright, time to roll up our sleeves and build the 'engine' for our web applications! We chose **Node.js** in the last chapter because it lets us use the JavaScript we already know. But how do we actually make a server with it? We'll use **Express.js**, the most popular framework for Node.js, to make our lives easier.

This chapter is all about getting from zero to a running backend server. We'll cover the theory behind each step and show you exactly how to write the code.

Node.js: JavaScript Everywhere üåç

Let's revisit: **Node.js** is a program you install on your computer. Its only job is to run JavaScript code *outside* of a web browser. Think about the JavaScript we wrote for the frontend ‚Äì it runs inside Chrome or Firefox. Node.js lets that *same* JavaScript language run directly on your computer or a server, allowing it to interact with the file system, databases, and network requests.

Key features of Node.js that make it popular for backends:

  • Asynchronous & Non-Blocking I/O:** This is Node's superpower. Imagine a waiter taking orders. A "blocking" waiter takes one order, goes to the kitchen, waits for the food, brings it back, and *only then* takes the next order. A "non-blocking" Node.js waiter takes an order, gives it to the kitchen, and immediately takes the next order without waiting. When the food is ready, the kitchen notifies the waiter. This makes Node.js extremely efficient at handling many user requests simultaneously (like in chat apps or APIs).
  • npm (Node Package Manager):** Node.js comes with npm, the world's largest library of pre-written code packages. Need to work with dates? There's a package. Need to connect to a database? There's a package. This saves developers huge amounts of time.

Express.js: The Friendly Layer on Top üöó

While Node.js provides the engine (`http` module for creating servers), writing a full server with just Node.js is tedious. You have to manually parse URLs, handle different request methods (GET, POST), manage headers, etc.

**Express.js** is a minimal and flexible Node.js web application framework. It provides a thin layer of fundamental web application features, without obscuring Node.js features. It simplifies the process immensely.

Think of it like this: Node.js gives you the raw engine parts. Express gives you a pre-assembled chassis, steering wheel, and pedals, making it much easier to build and drive your application.

Building Our First Server: Step-by-Step Breakdown üõ†Ô∏è

Let's go through the code we wrote previously, explaining each part in more detail. You'll need Node.js installed. Open your terminal.

Step 1: Project Setup & `npm init`

Every Node.js project needs a central file to keep track of its details and dependencies (the external packages it uses). This file is `package.json`.

How to Perform:

  1. Create a new directory (folder) for your project: `mkdir my-server`
  2. Navigate into that directory: `cd my-server`
  3. Run the command: `npm init -y`

Theory: What does `npm init -y` do?

  • `npm`: Stands for Node Package Manager. It's the command-line tool used to manage packages and run scripts for your Node.js projects.
  • `init`: This command initializes a new Node.js project.
  • `-y`: This flag automatically says "yes" to all the default questions `npm init` asks (like project name, version, etc.), quickly creating a basic `package.json` file.

Your `package.json` file will look something like this initially:

{
  "name": "my-server",
  "version": "1.0.0",
  "description": "",
  "main": "index.js", 
  "scripts": {
    "test": "echo \"Error: no test specified\" && exit 1"
  },
  "keywords": [],
  "author": "",
  "license": "ISC"
}

The `"main": "index.js"` part tells Node.js which file to run by default if you just type `node .`, but we'll be explicitly running our file, so it's not critical right now.

Step 2: Installing Express & `npm install`

Now we need to add the Express framework to our project. We use npm to download it from the internet and save it locally for our project.

How to Perform:

In the same `my-server` directory, run:

npm install express

Theory: What does `npm install express` do?

  • `install`: This npm command downloads and installs packages.
  • `express`: This is the name of the package we want to install.
  • npm looks up "express" in the online npm registry.
  • It downloads the Express.js code and all the other packages *it* depends on.
  • It places all this code inside a new folder called `node_modules`. You should *never* manually edit files inside `node_modules`.
  • It adds `express` as a dependency in your `package.json` file:
    •   "dependencies": {
    "express": "^4.17.1" // Version might differ
  }
  • It also creates or updates `package-lock.json`, which records the *exact* versions of all installed packages, ensuring your project works consistently everywhere.

Step 3 & 4: Creating `server.js` and Writing Code

Now we write the actual server logic.

How to Perform:

  1. Create a new file named `server.js` in your `my-server` folder.
  2. Add the following code:
// Line 1: Import Express using 'require'
const express = require('express');

// Line 2: Create the Express application object
const app = express();

// Line 3: Define the Port
const PORT = 3000; // Standard port for development

// Line 4: Define the '/' Route Handler (Homepage)
app.get('/', (req, res) => {
  // Line 5: Send a response back to the browser
  res.send('Hello World from Express!'); 
});

// Line 6: Start the Server
app.listen(PORT, () => {
  // Line 7: Log a confirmation message to the console
  console.log(`Server listening on port ${PORT}`);
});

Theory: Breaking down `server.js`

  • Line 1: `const express = require('express');`
    • `require()`: This is Node.js's built-in function to import modules (packages or files). It's like `import` in frontend JavaScript, but `require` is the standard in traditional Node.js (CommonJS modules).
    • `'express'`: We're telling Node to find the 'express' package we installed (it looks inside `node_modules`).
    • `const express = ...`: The `require` function returns the main function/object exported by the Express package, and we store it in a constant named `express`.
  • Line 2: `const app = express();`
    • We are calling the `express()` function we just imported. This function creates and returns an "app" object. This `app` object represents our Express application and has all the methods we need to build our server (like `.get()`, `.post()`, `.listen()`, etc.).
  • Line 3: `const PORT = 3000;`
    • A web server needs to "listen" for incoming requests on a specific network "port" on the computer. Ports are like apartment numbers for different applications. Port `3000` is a common convention for local development servers.
    • (Advanced: `process.env.PORT || 3000` is often used. This tries to get the port from an environment variable first, which is useful for deployment platforms like Heroku, otherwise it defaults to 3000).
  • Line 4: `app.get('/', (req, res) => { ... });`
    • `app.get()`: This method tells our Express app how to handle incoming **GET** requests (the most common type, used when you type a URL in your browser).
    • `/`: This is the **path** or **route** we want to handle. `/` always refers to the root or homepage of the website.
    • `(req, res) => { ... }`: This is the **handler function** (also called a callback function). Express runs this function *every time* a GET request comes in for the `/` path. It automatically provides two important objects as arguments:
      • `req` (Request): An object containing information about the incoming request (like headers, query parameters, user data).
      • `res` (Response): An object with methods for sending a response back to the client (the browser).
  • Line 5: `res.send('Hello World from Express!');`
    • `res.send()`: This is one of the methods on the `res` object. It sends a simple text-based response back to the browser. Express automatically sets the correct headers (like `Content-Type: text/html`).
  • Line 6: `app.listen(PORT, () => { ... });`
    • `app.listen()`: This method starts the server. It tells the Express app to start listening for incoming connections on the specified `PORT`.
    • `() => { ... }`: This is another callback function. `app.listen` runs this function *once* when the server successfully starts listening.
  • Line 7: `console.log(...)`
    • We log a message to the *terminal* (not the browser) to confirm that the server has started and tell us which port it's using.

Step 5: Running the Server with `node`

How to Perform:

In your terminal (in the `my-server` directory), run:

node server.js

Theory: What does `node server.js` do?

  • `node`: This command invokes the Node.js runtime environment that you installed.
  • `server.js`: You are telling Node.js which JavaScript file you want it to execute.
  • Node.js reads `server.js`, executes the code line by line, imports Express, sets up the route, and starts the server with `app.listen()`. The server then keeps running in your terminal, waiting for requests.

Now, open your browser and navigate to `http://localhost:3000`. Your browser sends a GET request to `/` on your local machine's port 3000. Your running Node.js server receives it, the `app.get('/', ...)` handler runs, and `res.send()` sends the "Hello World" message back to your browser, which displays it.

To stop the server, go back to your terminal and press `Ctrl + C`.

Tip:** Every time you change your `server.js` file, you need to stop the server (`Ctrl + C`) and restart it (`node server.js`) to see the changes. Tools like `nodemon` can automate this restarting process during development.

Next Steps: Building More Routes and APIs

You've now successfully built and understood your first Node.js and Express server! From here, you'll learn how to:

  • Handle different HTTP methods (like `POST` for submitting forms).
  • Work with **route parameters** (e.g., `/users/:userId`).
  • Use **middleware** (functions that run before your route handlers, useful for logging or authentication).
  • Send JSON data to build **APIs** for your frontend.
  • Connect to **Databases** to store and retrieve information.

This foundation is crucial for building the powerful backend logic that modern web applications require.
